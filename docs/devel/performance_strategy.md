# QV パフォーマンス戦略

## 目的
本ドキュメントは、QV（DICOM + VTK Volume Rendering）の性能・応答性を改善しつつ、画質とクロスプラットフォーム安定性を両立するための開発方針を定義する。

## 1. 解決したい問題と対策

### 問題A: DICOM読込直後にUIが固まり、CPUスパイクが発生する
対策:
- DICOM読込、ボリューム準備、ヒストグラム/統計計算をUIスレッドから分離する。
- 長時間処理には進捗表示とキャンセルを提供する。
- ヒストグラムはサンプリングで即時表示し、全量統計は必要時にバックグラウンドで実行する。

### 問題B: GPU mapper選択時でも操作が重い
対策:
- `PerformanceProfile`（`speed`/`balanced`/`quality`）を導入し、描画パラメータを集中管理する。
- 操作中は低品質、停止後に高品質へ戻す動的品質制御を導入する。
- 初期Transfer Functionを最適化し、過度に広い不透明領域によるレイマーチング負荷を抑える。

### 問題C: macOS/Linux/ドライバ差で体感性能が大きくぶれる
対策:
- OpenGL情報（vendor/renderer/version、direct rendering、mapper選択）をログ出力する。
- 実行環境に応じた初期プロファイル自動選択を導入する。
- 共通フォーマットで主要メトリクスを記録し、環境差を見える化する。

### 問題D: クリッピング処理が重く、操作体験を損なう
対策:
- 2段階処理（軽量プレビュー -> 確定時フル解像度適用）にする。
- プレビュー時は低解像度化またはROI制限を活用する。
- 現在の圧縮マスクUndo/Redoは維持しつつ、重い適用処理を非同期化する。

### 問題E: 改善効果を定量的に判断しにくい
対策:
- 性能KPIと受け入れ基準を定義する。
- 固定データセットでベンチマークを実施する。
- 性能変更PRにKPIのBefore/After比較を必須化する。

## 2. 設計方針

### 2.1 PerformanceProfileを中核概念として扱う
1つのプロファイルで以下を一括制御する:
- `ShadeOn/Off`
- サンプル距離と操作中オーバーライド
- Transfer Functionプリセット
- 必要に応じた品質フォールバック挙動

効果:
- チューニングの単一責務化
- 環境別調整の容易化
- 再現性のある動作

### 2.2 責務ごとのパイプライン分離
- `DataPipeline`: DICOM読込、正規化、基本統計
- `RenderPipeline`: mapper/property/volume更新、描画スケジューリング
- `InteractionPipeline`: カメラインタラクション時の品質切替
- `MaskPipeline`: クリッピングマスク生成、プレビュー、確定

UI層は重処理を直接持たず、状態制御と表示に専念する。

### 2.3 非同期実行を原則化
- UIスレッドで重処理を実行しない。
- 長時間処理はキャンセル可能にする。
- 非同期結果は「最新リクエスト優先」で古い結果を破棄する。

## 3. 実装ロードマップ

### Phase 1（最優先）
1. `PerformanceProfile` の実装
2. インタラクション中の動的品質切替
3. ヒストグラム初期表示のサンプリング経路を強制
4. KPIログ追加（`load_ms`, `first_frame_ms`, `fps_interaction`, `mask_apply_ms`）

### Phase 2
1. DICOM読込と統計計算の非同期化
2. 進捗表示 + キャンセル対応
3. 実行環境ベースの初期プロファイル自動選択

### Phase 3
1. クリッピング2段階処理（軽量プレビュー + フル確定）
2. クリッピング適用/復元の安全な非同期化
3. データサイズ別チューニングポリシーの精緻化

## 4. 性能KPIと受け入れ基準

`balanced` プロファイルでの基準例:
- 読込後の初回実用フレーム表示: 3.0秒以内（基準データ）
- ドラッグ操作中FPS: 20以上
- クリッピング確定待ち時間: 1.5秒以内（基準データ）
- 読込/操作中にUI無応答を発生させない
- 実行環境情報とKPIログを常時出力する

## 5. 開発・レビュー運用ルール

- 性能関連PRはKPIのBefore/After比較を必須とする。
- 1PRで扱う主要最適化テーマは1つに限定する。
- 応答性だけでなく視認性（画質）も必ず検証する。
- macOS/Linuxで最低限の回帰確認を行う。

## 6. 即時アクション項目

1. `PerformanceProfile` クラスとプリセット追加
2. `VolumeViewer.set_profile(profile)` を追加し設定適用を集約
3. インタラクション開始/終了イベントで品質切替
4. 構造化KPIログヘルパ追加
5. 固定ベンチマークデータと計測手順の文書化

## 7. 新たな優先順位（現状運用を踏まえた実装順）

前提:
- 通常のCT（例: 約200枚）では実用性能を満たしている。
- 問題は巨大データ（例: 1枚あたり非常に大きい画像）で顕在化しやすい。

優先順位:
1. ログ・可観測性の維持強化（OpenGL情報、mapper選択、KPI）
2. `PerformanceProfile` の枠組み導入（まずは `balanced` 固定でも可）
3. 大規模データ向けガードレール整備（推奨サイズ明記、警告表示、軽量モード案内）
4. 非同期DICOM読込の実装（再発時または機能拡張時に着手）
5. クリッピング2段階処理と重処理非同期化（高負荷案件が増えた段階で着手）

採用理由:
- 現在の主要ユースケースでの生産性を維持しつつ、将来の高負荷拡張に備える。
- 先に可観測性を強化することで、次の最適化投資を定量的に判断できる。
