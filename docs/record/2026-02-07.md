# 現在の問題整理（2026-02-07時点）

## 背景
- SIGSEGV が発生するため、VTK/Qt の描画経路を段階的に切り分け。
- VolumeViewer 単体は起動できるが、SliceViewerWidget 追加後に落ち始めた。

---

## 判明している事実

### 1. `Render()` がクラッシュ要因
- `showEvent` 由来の自動 `Render()` で確実に落ちる。
- `Render()` を抑止すると起動は安定する。

### 2. SliceViewerWidget の OpenGL 経路も不安定
- `Enable Slice Rendering` → `Render Slice Now` で落ちる。
- OpenGL/VTK の描画経路自体がこの環境では不安定。

### 3. VolumeViewer も `Render()` で落ちる
- `Enable Volume Rendering` → `Render Volume Now` で落ちる。
- `enable_rendering()` 内の `Render()` がクラッシュポイント。

---

## 切り分けの結果

| 項目 | 結果 |
|------|------|
| showEvent の自動 Render | 落ちる |
| Render を抑止 | 起動は安定 |
| QVTKOpenGLNativeWidget | 改善なし |
| ソフトウェア OpenGL | 改善なし |
| QSurfaceFormat(samples=0) | 改善なし |
| CPU Volume Mapper | 改善なし（Renderで落ちる） |
| SliceViewerWidget を別ウィンドウ | 依然落ちる |

---

## 現在の対応（安定化済み）

### SliceViewerWidget
- **OpenGL を使わない画像表示方式に変更**  
  - `vtkImageReslice` 出力 → `QImage` → `QLabel` 表示
  - **表示は成功**
- VTK RenderWindow / interactor は使用していない

### VolumeViewer
- `Render()` で落ちるため、描画経路は未解決
- interactor 初期化や Render を避けると起動は可能

---

## 現状の課題

1. **VolumeViewer の描画が依然クラッシュ**
   - Render呼び出し時点で落ちる
   - mapperや設定変更では回避できていない

2. **SliceViewerWidget は表示できるが操作が限定**
   - VTK interactor を使わないためホイールなど未対応
   - Qtイベントで再実装が必要

---

## 次の候補方針

### A. SliceViewerWidget の操作復活（Qt側で実装）
- ホイールでスライス移動
- 画像更新だけ行う（RenderWindow不要）

### B. VolumeViewer の描画経路再検討
- RenderWindowを使わない別経路（オフスクリーン＋QImage）
- もしくは環境側で VTK/Qt/GPU の組合せ変更

---

## まとめ
- **Render() がクラッシュの本体**
- Slice は OpenGL 回避で表示可能
- Volume は Render を使わない描画経路を検討する必要あり
